FIX Message Structure

8=FIX.4.4|9=122|35=D|34=2|49=BUYER|56=SELLER|11=12345|55=AAPL|54=1|38=1000|40=2|44=150.25|59=0|10=128|

Message breakdown:
8: BeginString (FIX version)
35: MsgType (D = New Order Single)
11: ClOrdID (Client Order ID)
55: Symbol (AAPL)
54: Side (1 = Buy)
38: OrderQty (1000 shares)
40: OrdType (2 = Limit)
44: Price ($150.25)
59: TimeInForce (0 = Day)

FIX Optimization Techniques

Binary FIX (FAST)

Use FAST (FIX Adapted for Streaming) for market data. Compresses messages 10-100x compared to text FIX for high-frequency data.

Session Configuration

Increase heartbeat interval to 60+ seconds for stable connections. Use sequence number reset at end of day to reduce recovery time.

Message Batching

Batch multiple orders in single message using repeating groups. Reduces message overhead and improves throughput for complex orders.

Choose FIX Engine Wisely

QuickFIX for flexibility, OnixS for low-latency, B2BITS for exchange connectivity. Open-source vs commercial trade-off of cost versus performance.

FIX Protocol Trade-offs

Advantages

  • -Industry standard with wide adoption
  • -Rich message set for complex workflows
  • -Standardized session management
  • -Support for allocations and regulatory reporting
  • -Proven reliability over decades

Limitations

  • -Higher latency than proprietary binary protocols
  • -Text-based parsing overhead
  • -Complex to implement full specification
  • -Vendor interpretations vary
  • -Session state management complexity

FIX Session Capacity Model

Session throughput T = N × (M / (R + P)) where N = network bandwidth, M = average message size, R = recovery capacity, P = processing capacity. For 1Gbps, 1KB messages, 80% utilization: 100,000 messages/second theoretical, 10,000-20,000 practical with resend recovery overhead.

FIX Session Recovery Logic

def handle_sequence_gap(session, received_seq, expected_seq):
    if received_seq > expected_seq:
        # Request missing messages
        request = f"35=2|34={session.next_sender_seq}|52={timestamp()}|7={expected_seq}|16={received_seq - expected_seq}|10=XXX|"
        session.send(request)
        session.state = 'RESEND_REQUEST'
    
    elif received_seq < expected_seq:
        # Duplicate message, ignore
        session.log.warning(f"Duplicate sequence {received_seq}, expected {expected_seq}")
        return None
    
    return True

Frequently Asked Questions

Which FIX version should I implement?

Start with FIX 4.4 (most widely supported). Add FIX 5.0 features only if needed for complex instruments (options, FX) or FIXatdl for algorithmic parameters.

How to handle FIX session disconnection?

Implement exponential backoff (1s, 2s, 4s, 8s, 16s). Use sequence number reset after prolonged disconnect. Request missing messages via ResendRequest (MsgType 2).

What's the difference between FIX and FIX Latest?

FIX 5.0 adds FIXatdl (algorithm parameters), FIXML (XML-based), and improved support for derivatives. Backward compatibility allows 4.4 clients to connect to 5.0 servers.
